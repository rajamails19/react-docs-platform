# Context Patterns

Once you understand the basics of `useContext`, these patterns help you build scalable, performant context-based state that doesn't re-render everything on every change.

## Pattern 1: Split Context for Performance

A single large context re-renders all consumers when any value changes. Split by update frequency:

```tsx
// ❌ One big context — changing theme re-renders UserPanel
const AppContext = createContext({ user, theme, notifications })

// ✅ Split — each context only affects its consumers
const UserContext    = createContext<User | null>(null)
const ThemeContext   = createContext<'light' | 'dark'>('dark')
const NotifContext   = createContext<Notification[]>([])

// Components subscribe to only what they need
function Avatar() {
  const user = useContext(UserContext)   // re-renders on user changes only
  return <img src={user?.avatar} />
}

function ThemeToggle() {
  const theme = useContext(ThemeContext)  // re-renders on theme changes only
  return <button>{theme}</button>
}
```

## Pattern 2: Stable Context Values with useMemo

```tsx
function UserProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null)

  // ❌ New object every render → all consumers re-render even when user didn't change
  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  )
}

// ✅ Memoize the value — only changes when user changes
function UserProvider({ children }: { children: React.ReactNode }) {
  const [user, setUser] = useState<User | null>(null)

  const value = useMemo(() => ({ user, setUser }), [user])

  return (
    <UserContext.Provider value={value}>
      {children}
    </UserContext.Provider>
  )
}
```

## Pattern 3: State + Dispatch Split

Separate read context from write context. Components that only read state don't re-render when dispatch changes:

```tsx
const CountStateContext   = createContext<number>(0)
const CountDispatchContext = createContext<React.Dispatch<Action>>(() => {})

function CountProvider({ children }: { children: React.ReactNode }) {
  const [count, dispatch] = useReducer(reducer, 0)

  return (
    <CountStateContext.Provider value={count}>
      <CountDispatchContext.Provider value={dispatch}>
        {children}
      </CountDispatchContext.Provider>
    </CountStateContext.Provider>
  )
}

// Display component — only needs state
function Display() {
  const count = useContext(CountStateContext)
  return <span>{count}</span>
}

// Button component — only needs dispatch, never re-renders on count change
function IncrementButton() {
  const dispatch = useContext(CountDispatchContext)
  return <button onClick={() => dispatch({ type: 'increment' })}>+</button>
}
```

## Pattern 4: Provider Composition

When you have many providers, compose them cleanly:

```tsx
// Without composition — pyramid of doom
function App() {
  return (
    <ThemeProvider>
      <AuthProvider>
        <NotificationProvider>
          <RouterProvider>
            <Main />
          </RouterProvider>
        </NotificationProvider>
      </AuthProvider>
    </ThemeProvider>
  )
}

// ✅ Compose into a single AppProviders component
function AppProviders({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider>
      <AuthProvider>
        <NotificationProvider>
          {children}
        </NotificationProvider>
      </AuthProvider>
    </ThemeProvider>
  )
}

function App() {
  return (
    <AppProviders>
      <Main />
    </AppProviders>
  )
}
```

## Pattern 5: Custom Hook with Error Boundary

Always wrap `useContext` in a custom hook that throws if used outside the Provider:

```tsx
const AuthContext = createContext<AuthContextValue | null>(null)

export function useAuth(): AuthContextValue {
  const ctx = useContext(AuthContext)
  if (!ctx) {
    throw new Error(
      'useAuth must be used within an AuthProvider. ' +
      'Wrap your component tree with <AuthProvider>.'
    )
  }
  return ctx
}

// Usage — clean, self-documenting, fails fast
function ProfileMenu() {
  const { user, logout } = useAuth()  // TypeScript knows this is non-null
  return <button onClick={logout}>{user.name}</button>
}
```

## Summary

<Callout type="success" title="What we learned">
- Split contexts by update frequency — don't put unrelated state in one context
- Memoize context values with `useMemo` to prevent unnecessary consumer re-renders
- Separate state and dispatch into two contexts — write-only components don't re-render on reads
- Compose multiple providers into one `AppProviders` component
- Always wrap `useContext` in a typed custom hook that validates the provider exists
</Callout>
