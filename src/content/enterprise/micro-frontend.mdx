# Micro Frontends

Micro frontends apply microservices principles to the frontend — splitting a large app into independently developed, tested, and deployed pieces owned by different teams.

## Webpack Module Federation

Module Federation (Webpack 5) lets applications load code from other applications at runtime:

```js
// webpack.config.js — HOST APPLICATION
const { ModuleFederationPlugin } = require('webpack').container

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'host',
      remotes: {
        // Load CheckoutApp from a different deployment
        checkout: 'checkout@https://checkout.example.com/remoteEntry.js',
        // Load CatalogApp from another deployment
        catalog:  'catalog@https://catalog.example.com/remoteEntry.js',
      },
      shared: {
        react:         { singleton: true, requiredVersion: '^18.0.0' },
        'react-dom':   { singleton: true, requiredVersion: '^18.0.0' },
      },
    }),
  ],
}
```

```js
// webpack.config.js — REMOTE APPLICATION (checkout team)
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'checkout',
      filename: 'remoteEntry.js',
      exposes: {
        './CheckoutFlow': './src/CheckoutFlow',
        './Cart':         './src/Cart',
      },
      shared: { react: { singleton: true } },
    }),
  ],
}
```

```tsx
// Host app — lazy load the remote component
const CheckoutFlow = React.lazy(() => import('checkout/CheckoutFlow'))
const Cart         = React.lazy(() => import('checkout/Cart'))

function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <Routes>
        <Route path="/checkout/*" element={<CheckoutFlow />} />
        <Route path="/cart"       element={<Cart />} />
      </Routes>
    </Suspense>
  )
}
```

## Shared Design System

A critical requirement: all micro frontends must share the same UI component library to look and feel consistent.

```ts
// packages/design-system/src/index.ts
export { Button }    from './Button'
export { Input }     from './Input'
export { Modal }     from './Modal'
export { theme }     from './theme'
export type { ButtonProps, InputProps } from './types'
```

```json
// Each micro frontend's package.json
{
  "dependencies": {
    "@company/design-system": "^2.0.0"
  }
}
```

<Callout type="warning" title="React Version Conflicts">
All micro frontends must use the same React version. Use `singleton: true` in Module Federation shared config. Multiple React instances on the same page cause hooks to fail with "Invalid hook call" errors.
</Callout>

## Cross-App Communication

```tsx
// Use a shared event bus — CustomEvents for simple cases
// Dispatch from one micro frontend
window.dispatchEvent(new CustomEvent('cart:item-added', {
  detail: { productId: '123', quantity: 1 }
}))

// Listen in another micro frontend
window.addEventListener('cart:item-added', (e: CustomEvent) => {
  updateCartBadge(e.detail.quantity)
})

// For complex state — use a shared Zustand store or Redux Toolkit
// in the design-system package that all apps import
```

## Summary

<Callout type="success" title="What we learned">
- Micro frontends apply microservices thinking to the frontend — independent teams, codebases, deploys
- Module Federation (Webpack 5) is the standard for runtime code sharing without rebuilding the host
- Shared `singleton: true` React prevents version conflicts across micro frontends
- A shared design system package is essential for visual consistency
- Use CustomEvents or a shared state store for cross-app communication
- Only adopt micro frontends when org size and team independence justify the overhead
</Callout>
