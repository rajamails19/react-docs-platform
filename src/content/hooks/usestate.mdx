# useState

`useState` is React's most fundamental hook. It lets you add **local state** to a functional component — state that persists between renders and triggers a re-render when it changes.

```tsx
const [state, setState] = useState(initialValue)
```

## The Problem It Solves

Without state, React components are pure functions: same input (props) → same output (JSX). But real UIs need to respond to user actions and remember information:

```tsx
// ❌ BROKEN — this never updates the UI
function Counter() {
  let count = 0   // regular variable, lost on every render
  return (
    <button onClick={() => count++}>  {/* increment has no effect */}
      Count: {count}
    </button>
  )
}

// ✅ WORKS — useState persists the value and triggers re-render
function Counter() {
  const [count, setCount] = useState(0)
  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  )
}
```

## Real-World Analogy

Think of state like a **whiteboard** in a classroom:

- The teacher (component) reads the number on the board (state)
- When the teacher erases and writes a new number (setState), everyone in the class sees the update
- The number persists between questions (renders) until explicitly changed

## Syntax & Patterns

### Basic usage

```tsx
const [count, setCount] = useState(0)
const [name, setName]   = useState('')
const [visible, setVisible] = useState(false)
```

### Setting state

```tsx
// Direct value
setCount(5)

// Functional updater (when new state depends on previous)
setCount(prev => prev + 1)   // ← always use this in event handlers

// Object state
const [user, setUser] = useState({ name: '', email: '' })

setUser({ name: 'Alice', email: 'alice@example.com' })     // full replace
setUser(prev => ({ ...prev, name: 'Bob' }))                // partial update
```

<Callout type="warning" title="Use functional updater for stale closures">
When the new state depends on the previous state, always use the functional form `setState(prev => ...)`. This guarantees you're working with the latest state, not a stale closure value — critical in async handlers and effects.
</Callout>

### Lazy initialization

If computing the initial state is expensive, pass a function:

```tsx
// ✅ Runs the function only once
const [data, setData] = useState(() => {
  const stored = localStorage.getItem('data')
  return stored ? JSON.parse(stored) : []
})

// ❌ Runs JSON.parse on EVERY render (wasteful)
const [data, setData] = useState(JSON.parse(localStorage.getItem('data') ?? '[]'))
```

## Multiple State Variables vs One Object

```tsx
// ✅ Preferred — separate variables for unrelated state
const [name, setName]       = useState('')
const [age, setAge]         = useState(0)
const [loading, setLoading] = useState(false)

// ✅ OK — one object when values always change together
const [position, setPosition] = useState({ x: 0, y: 0 })
const handleMouseMove = (e: MouseEvent) => {
  setPosition({ x: e.clientX, y: e.clientY })  // both change together
}

// ❌ Antipattern — grouping unrelated state into one object
const [state, setState] = useState({
  name: '',
  loading: false,
  data: [],
  error: null,
})
// Now every update requires spreading the whole object
setState(prev => ({ ...prev, loading: true }))  // tedious and error-prone
```

## State Immutability

React uses **reference equality** to detect state changes. Always return a new reference:

```tsx
const [todos, setTodos] = useState([
  { id: 1, text: 'Learn React', done: false }
])

// ❌ MUTATING — React won't see the change
const markDone = (id: number) => {
  todos.find(t => t.id === id)!.done = true   // mutates in place
  setTodos(todos)                              // same array reference → no re-render
}

// ✅ IMMUTABLE — new array reference triggers re-render
const markDone = (id: number) => {
  setTodos(prev =>
    prev.map(t => t.id === id ? { ...t, done: true } : t)
  )
}

// ✅ Adding to array
setTodos(prev => [...prev, { id: Date.now(), text: 'New task', done: false }])

// ✅ Removing from array
setTodos(prev => prev.filter(t => t.id !== id))
```

## Common Mistakes

| Mistake | Why it's wrong | Fix |
|---|---|---|
| Mutating state directly | React uses reference equality; same ref = no re-render | Always return new objects/arrays |
| Not using functional updater | Stale closure in async/batched updates | Use `prev => prev + 1` form |
| Storing derived data in state | Creates sync bugs and double source of truth | Compute from existing state |
| Initializing state from props | State doesn't update when props change | Use props directly or `useEffect` |
| Setting state in render | Creates infinite render loop | Only set state in handlers or effects |

## Beginner → Enterprise Patterns

### Beginner: Toggle

```tsx
const [open, setOpen] = useState(false)
<button onClick={() => setOpen(v => !v)}>Toggle</button>
```

### Intermediate: Form state

```tsx
const [form, setForm] = useState({ email: '', password: '' })
const update = (field: keyof typeof form) => (e: React.ChangeEvent<HTMLInputElement>) =>
  setForm(prev => ({ ...prev, [field]: e.target.value }))

<input value={form.email} onChange={update('email')} />
```

### Advanced: Optimistic updates

```tsx
const [todos, setTodos] = useState(serverTodos)

const deleteTodo = async (id: number) => {
  // Optimistically remove before API call
  setTodos(prev => prev.filter(t => t.id !== id))
  try {
    await api.deleteTodo(id)
  } catch {
    // Roll back on failure
    setTodos(serverTodos)
    toast.error('Failed to delete')
  }
}
```

## Summary

<Callout type="success" title="What we learned">
- `useState` stores a **persistent value** that survives re-renders and triggers a re-render when changed
- Always use the **functional updater** `setState(prev => ...)` when new state depends on previous
- Use **lazy initialization** `useState(() => computeExpensive())` for expensive initial values
- State is **immutable** — always return new references for objects and arrays
- Keep state **minimal** — derive everything else from it
</Callout>
