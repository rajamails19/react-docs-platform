# Vite Setup

**Vite** (French for "fast") is the recommended build tool for new React projects. It replaces Create React App and Webpack for most use cases, offering dramatically faster cold starts and Hot Module Replacement (HMR).

<Callout type="tip" title="Why Vite over CRA?">
Create React App uses Webpack, which bundles your entire app on startup. Vite uses native ES modules — the browser loads files on demand. A CRA app might take 30+ seconds to start; a Vite app starts in under a second.
</Callout>

## Prerequisites

Make sure you have Node.js 18+ installed:

```bash
node --version   # should print v18.x.x or higher
npm --version    # should print 9.x.x or higher
```

## Create a New Project

```bash
# Using npm
npm create vite@latest my-react-app -- --template react-ts

# Using yarn
yarn create vite my-react-app --template react-ts

# Using pnpm (recommended for monorepos)
pnpm create vite my-react-app --template react-ts
```

The `react-ts` template gives you React + TypeScript pre-configured.

## Start the Dev Server

```bash
cd my-react-app
npm install
npm run dev
```

You'll see:

```
  VITE v5.x.x  ready in 342 ms

  ➜  Local:   http://localhost:5173/
  ➜  Network: use --host to expose
  ➜  press h + enter to show help
```

<Callout type="success">
342ms startup time vs ~30-60 seconds with CRA. That's the Vite difference.
</Callout>

```
│   └── vite-env.d.ts      # Vite type declarations

├── vite.config.ts         # Vite configuration
├── tsconfig.json          # TypeScript config

```

## Key Configuration

```ts
// vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import path from 'path'

export default defineConfig({
  plugins: [
    react(),  // Enables Fast Refresh (HMR for React)
  ],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),  // Import @/components/...
    },
  },
  server: {
    port: 3000,
    open: true,  // Auto-open browser
  },
  build: {
    outDir: 'dist',
    sourcemap: true,
    rollupOptions: {
      output: {
        // Code splitting: vendor chunks cached by browsers
        manualChunks: {
          vendor: ['react', 'react-dom'],
        },
      },
    },
  },
})
```

## TypeScript Path Aliases

Add this to `tsconfig.json` to use `@/` imports:

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

Now you can import like this:

```tsx
// Instead of:
import { Button } from '../../../components/ui/Button'

// Write:
import { Button } from '@/components/ui/Button'
```

## Environment Variables

```bash
# .env.local (never commit this)
VITE_API_URL=https://api.example.com
VITE_GA_ID=G-XXXXXXXX

# .env (safe to commit — defaults only)
VITE_APP_NAME=My React App
```

```tsx
// Access in code with import.meta.env
const apiUrl = import.meta.env.VITE_API_URL
console.log(import.meta.env.MODE)  // "development" or "production"
```

<Callout type="warning" title="Prefix with VITE_">
Only variables prefixed with `VITE_` are exposed to your client-side code. Never put secrets in VITE_ variables — they are visible in the browser.
</Callout>

## Production Build

```bash
npm run build
```

Output in `dist/`:
```
dist/
├── assets/
│   ├── index-Cx7ygL2H.js       # Main bundle (code-split)
│   ├── vendor-D8kY7p3m.js      # React + react-dom (cached by CDN)
│   └── index-BvxCd4eH.css      # Styles
└── index.html
```

<Callout type="info" title="Content-hash filenames">
Vite automatically adds content hashes to filenames (e.g., `index-Cx7ygL2H.js`). This enables **long-term browser caching** — browsers cache the file indefinitely, but get the new version immediately when content changes.
</Callout>

## Summary

<Callout type="success" title="What we learned">
- Vite starts in milliseconds using native ES modules instead of bundling
- Use `npm create vite@latest` with the `react-ts` template for TypeScript support
- Configure path aliases (`@/`) for cleaner imports
- Environment variables must be prefixed with `VITE_`
- Production builds include code splitting and content-hash caching automatically
</Callout>
