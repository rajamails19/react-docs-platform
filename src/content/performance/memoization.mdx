# Memoization in React

Memoization is a caching technique where you store the result of an expensive computation and return the cached result when the same inputs appear again. React ships three memoization primitives: `React.memo`, `useMemo`, and `useCallback`. Used correctly they eliminate wasted renders. Used carelessly they add memory pressure and complexity with no measurable benefit.

<Callout type="warning" title="Profile before you memoize">
The biggest memoization mistake is applying it everywhere "just in case." Memoization has a cost: memory allocation, reference comparisons on every render, and cognitive overhead for future maintainers. Always profile with React DevTools first to confirm a real problem exists before optimizing.
</Callout>

## What Is Memoization?

At its core, memoization is a lookup table keyed on inputs:

```ts
// Conceptual implementation
function memoize<T>(fn: (...args: unknown[]) => T) {
  const cache = new Map<string, T>()
  return (...args: unknown[]): T => {
    const key = JSON.stringify(args)
    if (cache.has(key)) return cache.get(key)!
    const result = fn(...args)
    cache.set(key, result)
    return result
  }
}
```

React's memoization primitives work the same way — they skip expensive work when inputs have not changed since the last render.

## React.memo — Component Memoization

`React.memo` is a higher-order component that wraps a function component. Before rendering the child, React shallowly compares every prop. If all props are referentially equal to the previous render, the child's render function is skipped entirely.

```tsx
import { memo } from 'react'

interface ProductCardProps {
  id: string
  name: string
  price: number
  onAddToCart: (id: string) => void
}

// Without memo: re-renders whenever parent renders
// With memo: re-renders only when props actually change
const ProductCard = memo(function ProductCard({
  id,
  name,
  price,
  onAddToCart,
}: ProductCardProps) {
  console.log(`Rendering ProductCard ${id}`)

  return (
    <article className="product-card">
      <h3>{name}</h3>
      <p>${price.toFixed(2)}</p>
      <button onClick={() => onAddToCart(id)}>Add to Cart</button>
    </article>
  )
})

export default ProductCard
```

### Custom Comparison Function

`React.memo` accepts an optional second argument — a comparison function with signature `(prevProps, nextProps) => boolean`. Return `true` to skip the render, `false` to allow it.

```tsx
const ProductCard = memo(
  function ProductCard({ id, name, price, onAddToCart }: ProductCardProps) {
    return (/* ... */)
  },
  (prev, next) => {
    // Only re-render if price or name changed — ignore onAddToCart changes
    return prev.price === next.price && prev.name === next.name
  }
)
```

<Callout type="danger" title="Custom comparators hide bugs">
Custom comparison functions that ignore certain props (like `onAddToCart` above) can silently break behavior when the ignored prop changes. Prefer fixing the prop stability with `useCallback` instead of papering over it with a custom comparator.
</Callout>

## The Reference Equality Problem

This is the single most important concept for understanding why memoization often fails in practice.

JavaScript uses **reference equality** for objects and arrays. Two objects with identical content are not equal unless they are the same object in memory:

```ts
// Primitive equality — same value = same reference
42 === 42        // true
'hello' === 'hello'  // true

// Object equality — same shape ≠ same reference
{ a: 1 } === { a: 1 }   // false ← different objects in memory
[1, 2, 3] === [1, 2, 3] // false ← different arrays in memory
```

This breaks `React.memo` in a subtle way:

```tsx
function ProductList() {
  const [filter, setFilter] = useState('')

  // ❌ New object created on EVERY render — memo is useless
  const style = { color: 'red', fontSize: 14 }

  // ❌ New function created on EVERY render — memo is useless
  const handleAddToCart = (id: string) => {
    console.log('Adding', id)
  }

  return (
    <ProductCard
      id="abc"
      name="Widget"
      price={9.99}
      style={style}          // new reference every render
      onAddToCart={handleAddToCart} // new reference every render
    />
  )
}
```

Even though `ProductCard` is wrapped in `React.memo`, it re-renders on every parent render because `style` and `handleAddToCart` are new references each time.

## useMemo — Caching Computed Values

`useMemo` memoizes the **result** of a computation. It recalculates only when its dependency array changes.

```tsx
import { useMemo } from 'react'

function ProductList({ products, searchQuery, sortField }: Props) {
  // ✅ Only recomputes when products, searchQuery, or sortField changes
  const filteredAndSorted = useMemo(() => {
    console.log('Recomputing filtered list...')
    return products
      .filter(p =>
        p.name.toLowerCase().includes(searchQuery.toLowerCase())
      )
      .sort((a, b) => {
        if (sortField === 'price') return a.price - b.price
        return a.name.localeCompare(b.name)
      })
  }, [products, searchQuery, sortField])

  return (
    <ul>
      {filteredAndSorted.map(p => (
        <ProductCard key={p.id} {...p} />
      ))}
    </ul>
  )
}
```

### When useMemo Is Worth It

`useMemo` makes sense when:

1. The computation is measurably slow (verify with `performance.mark`)
2. The result is passed as a prop to a `React.memo` component (stabilizes the reference)
3. The result is used as a dependency in another `useMemo` or `useEffect`

```tsx
// Good use case: expensive computation
const priceHistogram = useMemo(() => {
  return computeHistogram(transactions) // O(n log n), n = 50,000
}, [transactions])

// Bad use case: trivial operation with unnecessary complexity
const doubled = useMemo(() => value * 2, [value]) // Just write: value * 2
```

## useCallback — Caching Function References

`useCallback` is `useMemo` for functions. It returns the same function reference across renders as long as dependencies haven't changed.

```tsx
import { useCallback, useState } from 'react'

function ShoppingCart({ productIds }: { productIds: string[] }) {
  const [cartItems, setCartItems] = useState<string[]>([])

  // ❌ New function every render — breaks React.memo on children
  const handleAdd = (id: string) => {
    setCartItems(prev => [...prev, id])
  }

  // ✅ Same function reference until dependencies change
  const handleAdd = useCallback((id: string) => {
    setCartItems(prev => [...prev, id])
  }, []) // No deps — setCartItems is stable

  // ✅ When you need a value from outside
  const handleRemove = useCallback((id: string) => {
    setCartItems(prev => prev.filter(item => item !== id))
  }, []) // setCartItems from useState is always stable

  return (
    <>
      {productIds.map(id => (
        <ProductCard key={id} id={id} onAddToCart={handleAdd} />
      ))}
    </>
  )
}
```

### useCallback vs useMemo Equivalence

They are fundamentally the same hook — `useCallback(fn, deps)` is sugar for `useMemo(() => fn, deps)`:

```tsx
// These are equivalent:
const stableCallback = useCallback((x: number) => x * 2, [])
const stableCallback = useMemo(() => (x: number) => x * 2, [])
```

## Combining All Three Correctly

The pattern that actually works: parent uses `useMemo` for object/array props and `useCallback` for function props, child is wrapped in `React.memo`.

```tsx
// types.ts
interface Column<T> {
  key: keyof T
  label: string
  render?: (value: T[keyof T], row: T) => React.ReactNode
}

interface DataTableProps<T> {
  rows: T[]
  columns: Column<T>[]
  onRowClick: (row: T) => void
  selectedIds: Set<string>
}

// DataTable.tsx — the memoized child
const DataTable = memo(function DataTable<T extends { id: string }>({
  rows,
  columns,
  onRowClick,
  selectedIds,
}: DataTableProps<T>) {
  return (
    <table>
      <thead>
        <tr>
          {columns.map(col => (
            <th key={String(col.key)}>{col.label}</th>
          ))}
        </tr>
      </thead>
      <tbody>
        {rows.map(row => (
          <tr
            key={row.id}
            onClick={() => onRowClick(row)}
            className={selectedIds.has(row.id) ? 'selected' : ''}
          >
            {columns.map(col => (
              <td key={String(col.key)}>
                {col.render
                  ? col.render(row[col.key], row)
                  : String(row[col.key])}
              </td>
            ))}
          </tr>
        ))}
      </tbody>
    </table>
  )
}) as <T extends { id: string }>(props: DataTableProps<T>) => React.ReactElement

// ParentPage.tsx — stabilizes all props for DataTable
function InventoryPage() {
  const { data: products } = useProducts()
  const [search, setSearch] = useState('')
  const [selectedIds, setSelectedIds] = useState<Set<string>>(new Set())

  // ✅ useMemo: stable array reference for columns (defined outside render ideally)
  const columns = useMemo<Column<Product>[]>(
    () => [
      { key: 'name', label: 'Product Name' },
      { key: 'sku', label: 'SKU' },
      {
        key: 'price',
        label: 'Price',
        render: (v) => `$${Number(v).toFixed(2)}`,
      },
      {
        key: 'stock',
        label: 'Stock',
        render: (v) => (
          <span className={Number(v) < 10 ? 'low-stock' : ''}>
            {String(v)}
          </span>
        ),
      },
    ],
    [] // Defined once — no external deps
  )

  // ✅ useMemo: filtered rows, stable reference
  const filteredProducts = useMemo(
    () =>
      products?.filter(p =>
        p.name.toLowerCase().includes(search.toLowerCase())
      ) ?? [],
    [products, search]
  )

  // ✅ useMemo: selectedIds as Set — new Set() on every render would break memo
  const selectedSet = useMemo(() => selectedIds, [selectedIds])

  // ✅ useCallback: stable function reference
  const handleRowClick = useCallback((product: Product) => {
    setSelectedIds(prev => {
      const next = new Set(prev)
      next.has(product.id) ? next.delete(product.id) : next.add(product.id)
      return next
    })
  }, [])

  return (
    <div>
      <input
        value={search}
        onChange={e => setSearch(e.target.value)}
        placeholder="Search products..."
      />
      <DataTable
        rows={filteredProducts}   // stable via useMemo
        columns={columns}         // stable via useMemo
        onRowClick={handleRowClick} // stable via useCallback
        selectedIds={selectedSet} // stable reference
      />
    </div>
  )
}
```

<Callout type="tip" title="Define static data outside the component">
If a value (like `columns` above) has no external dependencies, define it outside the component entirely — no `useMemo` needed, and it is created exactly once.

```tsx
// Best: outside the component, zero cost
const COLUMNS: Column<Product>[] = [
  { key: 'name', label: 'Product Name' },
  // ...
]

function InventoryPage() {
  // COLUMNS is always the same reference
  return <DataTable columns={COLUMNS} ... />
}
```
</Callout>

## The Cost of Over-Memoization

Memoization is not free. Each `useMemo`/`useCallback` call:

1. **Allocates memory** for the cached value and dependency array
2. **Runs a comparison loop** over the dependency array on every render
3. **Increases cognitive load** — readers must understand why the memo exists

```tsx
// ❌ Anti-pattern: memoizing everything reflexively
function UserProfile({ userId }: { userId: string }) {
  // useMemo on a trivial string concat — the comparison is slower than the work
  const displayName = useMemo(
    () => `User #${userId}`,
    [userId]
  )

  // useCallback with no child components to benefit
  const handleClick = useCallback(() => {
    console.log(userId)
  }, [userId])

  // useMemo wrapping a value that's already stable
  const config = useMemo(() => ({ theme: 'dark' }), [])
  // ↑ Better: const config = { theme: 'dark' } defined outside

  return <div onClick={handleClick}>{displayName}</div>
}
```

## Practical Example: Large Filterable Table

### Before Optimization

```tsx
// BEFORE — Every keystroke re-renders all 500 rows
function EmployeeDirectory() {
  const [search, setSearch] = useState('')
  const [dept, setDept] = useState('all')
  const employees = useEmployees() // 500 records

  // New array reference every render
  const filtered = employees.filter(e =>
    e.name.toLowerCase().includes(search.toLowerCase()) &&
    (dept === 'all' || e.department === dept)
  )

  // New function every render
  const handleSelect = (id: string) => {
    console.log('Selected', id)
  }

  return (
    <div>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <select value={dept} onChange={e => setDept(e.target.value)}>
        {/* options */}
      </select>
      {filtered.map(emp => (
        // Not memoized, receives new handleSelect every render
        <EmployeeRow key={emp.id} employee={emp} onSelect={handleSelect} />
      ))}
    </div>
  )
}
```

**Profile result:** 500 × `EmployeeRow` re-renders on every keystroke. Total: ~120ms per keypress — noticeably janky.

### After Optimization

```tsx
// AFTER — Only changed rows re-render
const EmployeeRow = memo(function EmployeeRow({
  employee,
  onSelect,
}: {
  employee: Employee
  onSelect: (id: string) => void
}) {
  return (
    <div className="employee-row" onClick={() => onSelect(employee.id)}>
      <img src={employee.avatar} alt={employee.name} />
      <span>{employee.name}</span>
      <span>{employee.department}</span>
      <span>{employee.role}</span>
    </div>
  )
})

function EmployeeDirectory() {
  const [search, setSearch] = useState('')
  const [dept, setDept] = useState('all')
  const employees = useEmployees()

  // Stable filtered array — only recomputes when inputs change
  const filtered = useMemo(
    () =>
      employees.filter(
        e =>
          e.name.toLowerCase().includes(search.toLowerCase()) &&
          (dept === 'all' || e.department === dept)
      ),
    [employees, search, dept]
  )

  // Stable function reference — child memo can now bail out
  const handleSelect = useCallback((id: string) => {
    console.log('Selected', id)
  }, [])

  return (
    <div>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <select value={dept} onChange={e => setDept(e.target.value)}>
        {/* options */}
      </select>
      {filtered.map(emp => (
        <EmployeeRow key={emp.id} employee={emp} onSelect={handleSelect} />
      ))}
    </div>
  )
}
```

**Profile result after:** Only the rows whose data actually changed re-render. On a typical search keystroke, 0–3 rows re-render. Total: ~2ms per keypress.

## startTransition as an Alternative

Sometimes memoization is the wrong tool. If a state update triggers a slow render that you cannot avoid (e.g., rendering many new items), `startTransition` keeps the UI responsive by marking the update as non-urgent.

```tsx
import { useState, startTransition } from 'react'

function SearchPage() {
  const [query, setQuery] = useState('')
  const [results, setResults] = useState<Result[]>([])

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value

    // ✅ Input update is urgent — happens synchronously
    setQuery(value)

    // ✅ Results update is non-urgent — React may interrupt it
    startTransition(() => {
      setResults(searchProducts(value)) // expensive but deferrable
    })
  }

  return (
    <>
      <input value={query} onChange={handleChange} />
      {/* Results render is deferred — input stays responsive */}
      <ResultsList results={results} />
    </>
  )
}
```

## Common Mistakes and Fixes

### Mistake 1: Object in Dependency Array

```tsx
// ❌ options is a new object every render → infinite effect loop
function Chart({ data, options }: Props) {
  useEffect(() => {
    drawChart(data, options)
  }, [data, options]) // options = {} is always new
}

// ✅ Destructure the primitive values instead
function Chart({ data, options }: Props) {
  const { color, showLegend } = options
  useEffect(() => {
    drawChart(data, { color, showLegend })
  }, [data, color, showLegend]) // primitives — stable
}
```

### Mistake 2: Missing Dependencies in useCallback

```tsx
// ❌ Stale closure — userId is captured from first render
const fetchUser = useCallback(() => {
  api.getUser(userId) // userId might be stale
}, []) // Missing dependency!

// ✅ Include all dependencies
const fetchUser = useCallback(() => {
  api.getUser(userId)
}, [userId])
```

### Mistake 3: Wrapping Everything in React.memo

```tsx
// ❌ These components re-render so rarely that memo adds no value
// and adds visual noise + memory overhead
const PageTitle = memo(({ title }: { title: string }) => <h1>{title}</h1>)
const Divider = memo(() => <hr />)
const Badge = memo(({ count }: { count: number }) => <span>{count}</span>)

// ✅ Only memo components that:
// - Render often due to parent re-renders
// - Have expensive render logic
// - Receive stable (or stabilized) props
```
