# Nested Routes in React Router v6

Real applications are rarely flat. A dashboard has a sidebar. A settings page has tabs. A profile section has sub-pages. React Router's nested route system lets you map this natural hierarchy directly into your URL structure and component tree.

<Callout type="info">
Nested routes were fundamentally redesigned in React Router v6. If you're coming from v5, the `<Switch>` and manually nested `<Route>` patterns no longer apply. Everything now uses the `<Routes>` tree and the `<Outlet>` component.
</Callout>

## What Are Nested Routes?

A nested route is a route that lives _inside_ another route in both the URL and the component hierarchy. When you visit `/dashboard/settings`, the router renders both the `Dashboard` layout component **and** the `Settings` component at the same time — the parent wraps the child.

This solves two real problems:

1. **Shared UI** — The sidebar, header, and navigation that appear on every dashboard page only need to be defined once in the parent layout.
2. **URL hierarchy** — The URL structure mirrors your UI structure, making the app predictable and bookmarkable.

## The `Outlet` Component

`<Outlet>` is the single most important concept for nested routes. It acts as a **placeholder** inside a parent route component where the matched child route's component will be rendered.

```tsx
// layouts/DashboardLayout.tsx
import { Outlet, NavLink } from 'react-router-dom'

export function DashboardLayout() {
  return (
    <div className="dashboard">
      <aside className="sidebar">
        <nav>
          <NavLink to="/dashboard" end>Overview</NavLink>
          <NavLink to="/dashboard/analytics">Analytics</NavLink>
          <NavLink to="/dashboard/settings">Settings</NavLink>
          <NavLink to="/dashboard/profile">Profile</NavLink>
        </nav>
      </aside>
      <main className="content">
        {/* Child route renders HERE */}
        <Outlet />
      </main>
    </div>
  )
}
```

Without `<Outlet>`, the parent component renders but the child route's component never appears. This is the most common beginner mistake.

<Callout type="warning">
If you define child routes but see only the parent and nothing else, you almost certainly forgot to render `<Outlet />` in the parent component.
</Callout>

## Defining Nested Routes in JSX

Nesting routes in JSX is done by placing `<Route>` elements **inside** other `<Route>` elements. The parent's `path` becomes the prefix for all child paths.

```tsx
// App.tsx
import { BrowserRouter, Routes, Route } from 'react-router-dom'
import { DashboardLayout } from './layouts/DashboardLayout'
import { Overview } from './pages/dashboard/Overview'
import { Analytics } from './pages/dashboard/Analytics'
import { Settings } from './pages/dashboard/Settings'
import { Profile } from './pages/dashboard/Profile'
import { Home } from './pages/Home'
import { NotFound } from './pages/NotFound'

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />

        {/* Parent layout route */}
        <Route path="/dashboard" element={<DashboardLayout />}>
          {/* Index route — rendered at /dashboard */}
          <Route index element={<Overview />} />
          {/* Child routes — paths are relative to /dashboard */}
          <Route path="analytics" element={<Analytics />} />
          <Route path="settings" element={<Settings />} />
          <Route path="profile" element={<Profile />} />
        </Route>

        <Route path="*" element={<NotFound />} />
      </Routes>
    </BrowserRouter>
  )
}
```

Notice that child route paths (`"analytics"`, `"settings"`) are written **without** a leading slash. React Router automatically prepends the parent's path (`/dashboard`), resulting in `/dashboard/analytics`, `/dashboard/settings`, etc.

<Callout type="tip">
Child route paths should never start with `/`. A leading slash makes the path absolute, which will break the nesting and cause the route to be treated as a top-level route.
</Callout>

## Layout Routes (Parent with Shared UI)

A **layout route** is a parent route that exists purely to provide shared UI (header, sidebar, footer). It wraps all child routes. The parent route's `element` renders the shared structure, and `<Outlet>` is where child components slot in.

Layout routes can be _pathless_ — they share UI without adding a URL segment:

```tsx
// A pathless layout route — no path prop
<Route element={<AuthenticatedLayout />}>
  <Route path="/dashboard" element={<Dashboard />} />
  <Route path="/profile" element={<Profile />} />
  <Route path="/settings" element={<Settings />} />
</Route>
```

```tsx
// layouts/AuthenticatedLayout.tsx
import { Outlet, Navigate } from 'react-router-dom'
import { useAuth } from '../hooks/useAuth'
import { AppHeader } from '../components/AppHeader'
import { AppFooter } from '../components/AppFooter'

export function AuthenticatedLayout() {
  const { user } = useAuth()

  if (!user) {
    return <Navigate to="/login" replace />
  }

  return (
    <>
      <AppHeader user={user} />
      <main>
        <Outlet />
      </main>
      <AppFooter />
    </>
  )
}
```

This pattern lets you apply authentication checks, shared headers, or analytics tracking to entire sections of your app without repeating logic in every page component.

## Index Routes

An index route is the **default child** rendered when the parent's URL matches exactly and no other child route matches. You declare it with the `index` prop (no `path` needed).

```tsx
<Route path="/dashboard" element={<DashboardLayout />}>
  {/* Rendered at exactly /dashboard */}
  <Route index element={<Overview />} />
  <Route path="analytics" element={<Analytics />} />
</Route>
```

Without an index route, visiting `/dashboard` would render the layout's `<Outlet>` as empty — you'd see the sidebar but no content in the main area.

<Callout type="tip">
Every layout route that has child routes should also have an `index` route to handle the exact parent URL. Think of it as the "home page" for that section.
</Callout>

### Index Route vs. Empty Path

These look similar but behave differently:

```tsx
// Index route — only matches /dashboard exactly
<Route index element={<Overview />} />

// Empty string path — matches /dashboard/ (with trailing slash) too
<Route path="" element={<Overview />} />
```

Use `index` — it's the idiomatic React Router v6 approach.

## Relative Links Inside Nested Routes

Inside a nested route context, `<Link>` and `<NavLink>` paths can be **relative**. A relative path is resolved against the current route's location, similar to how relative file paths work.

```tsx
// pages/dashboard/Settings.tsx
// Currently at /dashboard/settings
import { Link } from 'react-router-dom'

export function Settings() {
  return (
    <div>
      <h1>Settings</h1>
      {/* Absolute link — always works, always explicit */}
      <Link to="/dashboard/profile">Go to Profile</Link>

      {/* Relative link — resolves to /dashboard/profile */}
      <Link to="../profile">Go to Profile</Link>

      {/* Relative link — resolves to /dashboard/settings/notifications */}
      <Link to="notifications">Notifications</Link>
    </div>
  )
}
```

The `..` syntax moves up one route segment. This is useful for keeping components portable — if you rename the parent path, relative links inside still work.

<Callout type="info">
Relative links in React Router v6 are route-relative, not URL-segment-relative. `..` goes up one **route** level, which may not always correspond to one URL segment if the route has a multi-segment path.
</Callout>

## `NavLink` with Active Styles in Nested Context

`NavLink` automatically applies an `active` class when the current URL matches its `to` prop. In nested contexts, use the `end` prop to prevent a parent link from staying active when a child route is active.

```tsx
// layouts/DashboardLayout.tsx
import { NavLink } from 'react-router-dom'

export function DashboardLayout() {
  const navLinkClass = ({ isActive }: { isActive: boolean }) =>
    `nav-link ${isActive ? 'nav-link--active' : ''}`

  return (
    <aside className="sidebar">
      <nav>
        {/*
          The `end` prop means this only matches /dashboard exactly.
          Without `end`, it would be active on ALL /dashboard/* routes.
        */}
        <NavLink to="/dashboard" className={navLinkClass} end>
          Overview
        </NavLink>
        <NavLink to="/dashboard/analytics" className={navLinkClass}>
          Analytics
        </NavLink>
        <NavLink to="/dashboard/settings" className={navLinkClass}>
          Settings
        </NavLink>
        <NavLink to="/dashboard/profile" className={navLinkClass}>
          Profile
        </NavLink>
      </nav>
      <main>
        <Outlet />
      </main>
    </aside>
  )
}
```

### App.tsx — Full Route Tree

```tsx
// App.tsx
import { BrowserRouter, Routes, Route } from 'react-router-dom'
import { DashboardLayout } from './layouts/DashboardLayout'
import { SettingsLayout } from './pages/dashboard/SettingsLayout'
import { Home } from './pages/Home'
import { Overview } from './pages/dashboard/Overview'
import { Analytics } from './pages/dashboard/Analytics'
import { GeneralSettings } from './pages/dashboard/GeneralSettings'
import { SecuritySettings } from './pages/dashboard/SecuritySettings'
import { Profile } from './pages/dashboard/Profile'

function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<Home />} />

        <Route path="/dashboard" element={<DashboardLayout />}>
          <Route index element={<Overview />} />
          <Route path="analytics" element={<Analytics />} />

          {/* Two levels of nesting: /dashboard/settings/general */}
          <Route path="settings" element={<SettingsLayout />}>
            <Route index element={<GeneralSettings />} />
            <Route path="security" element={<SecuritySettings />} />
          </Route>

          <Route path="profile" element={<Profile />} />
        </Route>

        <Route path="*" element={<div>Page not found</div>} />
      </Routes>
    </BrowserRouter>
  )
}

export default App
```

## Passing Data Through `Outlet` Context

Sometimes a parent route needs to pass data down to child routes without prop drilling. `<Outlet>` accepts a `context` prop that child routes can read with `useOutletContext`.

```tsx
// layouts/DashboardLayout.tsx
import { Outlet } from 'react-router-dom'

interface DashboardContextType {
  user: { id: string; name: string; role: string }
  refreshUser: () => void
}

export function DashboardLayout() {
  const [user, setUser] = useState({ id: '1', name: 'Alice', role: 'admin' })

  const refreshUser = async () => {
    const fresh = await fetchUser(user.id)
    setUser(fresh)
  }

  return (
    <div className="dashboard">
      <Sidebar />
      <main>
        {/* Pass context to all child routes */}
        <Outlet context={{ user, refreshUser } satisfies DashboardContextType} />
      </main>
    </div>
  )
}
```

```tsx
// pages/dashboard/Profile.tsx
import { useOutletContext } from 'react-router-dom'

interface DashboardContextType {
  user: { id: string; name: string; role: string }
  refreshUser: () => void
}

export function Profile() {
  const { user, refreshUser } = useOutletContext<DashboardContextType>()

  return (
    <div>
      <h1>{user.name}'s Profile</h1>
      <p>Role: {user.role}</p>
      <button onClick={refreshUser}>Refresh</button>
    </div>
  )
}
```

<Callout type="tip">
`useOutletContext` is great for data that is scoped to a layout section — like the logged-in user on authenticated pages — without reaching for global state management.
</Callout>

## `useParams` in Nested Routes

Dynamic segments work at any level of nesting. Params accumulate from all matched routes.

```tsx
// Route definition
<Route path="/users/:userId" element={<UserLayout />}>
  <Route index element={<UserProfile />} />
  <Route path="posts/:postId" element={<UserPost />} />
</Route>

// pages/UserPost.tsx
import { useParams } from 'react-router-dom'

export function UserPost() {
  // Both params are available even though :userId comes from the parent route
  const { userId, postId } = useParams<{ userId: string; postId: string }>()

  return <h1>Post {postId} by User {userId}</h1>
}
```

## Summary

- **Nested routes** map URL hierarchy to component hierarchy, enabling shared layouts across multiple pages.
- **`<Outlet>`** is the insertion point for child route components inside a parent — always required.
- **Layout routes** wrap child routes in shared UI. They can have a path (adding a URL segment) or be pathless (grouping without a URL segment).
- **Index routes** (`<Route index>`) render at the exact parent URL as the default child.
- **Child route paths** are relative — never use a leading `/` on them.
- **`NavLink` with `end`** prevents parent links from staying active on child route URLs.
- **`useOutletContext`** lets parent layouts pass data to child routes without prop drilling.
