# useReducer

`useReducer` is React's alternative to `useState` for managing **complex state logic**. It follows the same pattern as Redux: a pure **reducer function** transforms state based on dispatched **actions**.

```tsx
const [state, dispatch] = useReducer(reducer, initialState)
```

## When useState Isn't Enough

As state grows more complex, `useState` leads to scattered, hard-to-trace updates:

```tsx
// ❌ 5 separate state values that are logically connected
const [loading, setLoading]   = useState(false)
const [data, setData]         = useState<User[]>([])
const [error, setError]       = useState<string | null>(null)
const [page, setPage]         = useState(1)
const [total, setTotal]       = useState(0)

// Updating these in sync is error-prone:
setLoading(true)
setError(null)
// ...3 more setters per action
```

## The Reducer Pattern

```tsx
// ✅ All state in one place, all transitions explicit
type State = {
  loading: boolean
  data: User[]
  error: string | null
  page: number
  total: number
}

type Action =
  | { type: 'FETCH_START' }
  | { type: 'FETCH_SUCCESS'; payload: { data: User[]; total: number } }
  | { type: 'FETCH_ERROR';   payload: string }
  | { type: 'SET_PAGE';      payload: number }

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'FETCH_START':
      return { ...state, loading: true, error: null }
    case 'FETCH_SUCCESS':
      return { ...state, loading: false, data: action.payload.data, total: action.payload.total }
    case 'FETCH_ERROR':
      return { ...state, loading: false, error: action.payload }
    case 'SET_PAGE':
      return { ...state, page: action.payload }
    default:
      return state
  }
}
```

## Real-World Analogy

useReducer is like a **bank teller system**:
- **State** = your account balance
- **Actions** = deposit slip, withdrawal slip, transfer form
- **Reducer** = bank's processing rules
- **dispatch** = submitting a transaction

You don't directly change the balance. You submit a standardized form (action), and the bank's rules (reducer) decide what happens.

    { id: 3, text: 'Ship to prod',     done: false, priority: 'low' },
  ],
  filter: 'all',
  nextId: 4,
}

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'ADD':

        ...state,

## useReducer vs useState

| Scenario | useState | useReducer |
|---|---|---|
| Simple boolean toggle | ✅ Perfect | Overkill |
| Single string/number | ✅ Perfect | Overkill |
| Multiple related values | OK | ✅ Better |
| Complex update logic | Messy | ✅ Clean |
| Next state depends on multiple prev values | Error-prone | ✅ Explicit |
| Business rules / state machines | Hard | ✅ Natural |
| Testing state transitions | Hard | ✅ Easy (pure function) |

## Testing Reducers

Because reducers are pure functions, they're trivially testable:

```tsx
import { reducer, initialState } from './todoReducer'

test('adds a todo', () => {
  const state = reducer(initialState, { type: 'ADD', payload: 'Buy milk' })
  expect(state.todos).toHaveLength(1)
  expect(state.todos[0].text).toBe('Buy milk')
  expect(state.todos[0].done).toBe(false)
})

test('toggles a todo', () => {
  const withTodo = reducer(initialState, { type: 'ADD', payload: 'Task' })
  const toggled  = reducer(withTodo, { type: 'TOGGLE', payload: withTodo.todos[0].id })
  expect(toggled.todos[0].done).toBe(true)
})
```

## Summary

<Callout type="success" title="What we learned">
- `useReducer` manages complex state with explicit actions and a pure reducer function
- Perfect for state with multiple sub-values, complex transitions, or business rules
- Reducers are **pure functions** — same input always produces same output
- This makes state transitions **testable, auditable, and predictable**
- `dispatch` + action types create a clear audit trail of what changed and why
</Callout>
