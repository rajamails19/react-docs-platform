# SPA vs MPA

Before you write your first React component, it helps to understand the architectural world React lives in. React is designed for building **Single Page Applications (SPAs)** — but what does that mean, and how does it differ from the traditional web?

## What Is a Multi Page Application (MPA)?

A **Multi Page Application** is the classic model of the web. Every time you navigate to a new URL, the browser makes a full HTTP request to the server. The server returns a completely new HTML document, and the browser replaces the entire page.

Think of **Wikipedia**. Click any link and your browser:

1. Sends a GET request to the server
2. Receives a fresh HTML document
3. Discards the current page entirely
4. Parses and renders the new page from scratch

This is the default behavior the web was built on.

## What Is a Single Page Application (SPA)?

A **Single Page Application** loads a single HTML file once. After that initial load, navigation is handled entirely by JavaScript — the URL changes, the UI updates, but the browser never does a full page reload.

Think of **Gmail**. Click on an email, click Compose, navigate to Settings — the page never fully reloads. JavaScript intercepts your clicks, fetches only the data it needs (via API calls), and re-renders just the parts of the UI that changed.

<MermaidDiagram chart={`
sequenceDiagram
    participant Browser
    participant Server
    participant API

    Browser->>Server: GET / (initial load)
    Server-->>Browser: index.html + JS bundle
    Note over Browser: App boots, JS takes over routing

    Browser->>API: fetch /api/inbox (click Inbox)
    API-->>Browser: JSON data
    Note over Browser: JS updates UI — no page reload

    Browser->>API: fetch /api/compose (click Compose)
    API-->>Browser: JSON data
    Note over Browser: JS updates UI — no page reload
`} />

<Callout type="info" title="The URL still changes in SPAs">
Even though the page never reloads, SPAs update the URL using the browser's History API (`window.history.pushState`). This keeps URLs shareable and the back button functional. React Router manages this for you.
</Callout>

## React Is Built for SPAs

React's rendering model is fundamentally SPA-oriented. When you write:

```tsx
function App() {
  return (
    <Routes>
      <Route path="/" element={<Home />} />
      <Route path="/about" element={<About />} />
      <Route path="/dashboard" element={<Dashboard />} />
    </Routes>
  )
}
```

React Router intercepts clicks on `<Link>` components, updates the URL using the History API, and swaps out which component renders — all without a page reload. The server never sees these navigation events.

### How React Router Enables SPA Navigation

```tsx
import { BrowserRouter, Routes, Route, Link } from 'react-router-dom'

function App() {
  return (
    <BrowserRouter>
      {/* Navigation — no <a> tags, no page reloads */}
      <nav>
        <Link to="/">Home</Link>
        <Link to="/profile">Profile</Link>
        <Link to="/settings">Settings</Link>
      </nav>

      {/* Only the matched route renders */}
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/profile" element={<ProfilePage />} />
        <Route path="/settings" element={<SettingsPage />} />
      </Routes>
    </BrowserRouter>
  )
}
```

When a user clicks `<Link to="/profile">`, React Router:

1. Calls `event.preventDefault()` to stop the browser's default navigation
2. Calls `window.history.pushState({}, '', '/profile')` to update the URL
3. Re-renders `<Routes>`, which now matches `/profile` and renders `<ProfilePage />`

No server request. No flash. Instant.

## The Initial Load Trade-off

The biggest SPA disadvantage is the initial bundle size. On first visit, the browser must download, parse, and execute your entire JavaScript application before anything is interactive.

```
MPA — visiting /home:
  GET /home → 50KB HTML → instant content ✅

SPA — first visit:
  GET / → 5KB HTML + 300KB JS bundle → parse → execute → render → content ✅
  (but then all future navigation is instant)
```

This is why SPAs can feel slow on first load but fast afterward.

<Callout type="tip" title="Code splitting solves the bundle problem">
Modern React apps use **code splitting** with `React.lazy()` and dynamic `import()` to break the bundle into route-based chunks. Only the code needed for the current route is downloaded. This gives you SPA speed with MPA-like initial load times.
</Callout>

## The SEO Problem with SPAs

Search engine crawlers traditionally read the HTML they receive from the server. A pure SPA sends nearly empty HTML:

```html
<!-- What a crawler sees before JS runs -->
<!DOCTYPE html>
<html>
  <head><title>My App</title></head>
  <body>
    <div id="root"></div>  <!-- Empty! -->
    <script src="/bundle.js"></script>
  </body>
</html>
```

Modern solutions:

- **Server-Side Rendering (SSR)** — Next.js renders React on the server, sending full HTML
- **Static Site Generation (SSG)** — Pre-render pages at build time
- **Prerendering** — Use tools like `react-snap` to pre-render SPA routes

<Callout type="info" title="Googlebot can execute JavaScript">
Google's crawler does execute JavaScript now, but with a delay. Bing, social media crawlers (Twitter cards, Facebook OG), and many others still don't. For SEO-critical sites, use Next.js or Remix.
</Callout>

## Real-World Examples

### SPAs in the wild

| App | Why SPA makes sense |
|-----|---------------------|
| Gmail | Persistent sidebar, fast email switching, offline support |
| Figma | Complex canvas state that can't survive page reloads |
| Notion | Rich text editor with continuous autosave |
| Twitter/X | Real-time feed, infinite scroll, no interruptions |
| Trello | Drag-and-drop board state must persist across navigation |

### MPAs in the wild

| App | Why MPA makes sense |
|-----|---------------------|
| Wikipedia | Content-heavy, SEO-critical, simple navigation |
| Amazon product pages | Each page is independently cacheable, SEO essential |
| News sites | Article pages need full SEO, fast first contentful paint |
| Government sites | Accessibility requirements, broad browser support |

## When to Choose Which

**Choose SPA (React default) when:**
- Building a web *application* with complex user interactions
- Users will navigate frequently within a session
- You need app-like UX (no flash, smooth transitions)
- SEO is secondary (dashboards, tools, authenticated areas)
- You want to reuse a single API between web and mobile

**Choose MPA (or add SSR) when:**
- Building a content-heavy *website* (blog, e-commerce, news)
- SEO is critical for every page
- You need the fastest possible first contentful paint
- Your audience is on slow connections or older devices
- Each page is independently meaningful and shareable

**The hybrid approach** — frameworks like **Next.js** let you mix both: SSR for public/SEO pages, SPA behavior for authenticated areas. This is increasingly the industry default for large applications.

<Callout type="success" title="Key takeaway">
React is a SPA library by default, but it doesn't have to be SPA-only. Understanding this architecture helps you make smarter decisions about routing, data fetching, and choosing between create-react-app, Vite, Next.js, or Remix for your project.
</Callout>
