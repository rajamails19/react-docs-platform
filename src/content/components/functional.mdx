# Functional Components

A **functional component** is a JavaScript function that accepts `props` and returns JSX. Since React Hooks were introduced in 2019, functional components are the standard way to write React.

```tsx
function Greeting({ name }: { name: string }) {
  return <h1>Hello, {name}!</h1>
}
```

## Anatomy of a Functional Component

```tsx
// 1. TypeScript interface for props
interface CardProps {
  title:    string
  body:     string
  footer?:  React.ReactNode   // optional
  onClick?: () => void
}

// 2. Component function — named, starts with capital letter
function Card({ title, body, footer, onClick }: CardProps) {
  // 3. Hooks at the top level
  const [expanded, setExpanded] = useState(false)

  // 4. Derived values and handlers
  const handleClick = () => {
    setExpanded(v => !v)
    onClick?.()
  }

  // 5. Return JSX (one root element, or Fragment)
  return (
    <div className="card" onClick={handleClick}>
      <h2>{title}</h2>
      {expanded && <p>{body}</p>}
      {footer && <footer>{footer}</footer>}
    </div>
  )
}

// 6. Export — named or default
export default Card
```

## Arrow Function vs Named Function

```tsx
// Both are valid functional components

// Named function (recommended — better stack traces)
function Button({ label, onClick }: ButtonProps) {
  return <button onClick={onClick}>{label}</button>
}

// Arrow function (common in inline/HOC patterns)
const Button = ({ label, onClick }: ButtonProps) => (
  <button onClick={onClick}>{label}</button>
)
```

## TypeScript Props Typing

```tsx
// Method 1: Inline type annotation (quick, good for simple components)
function Tag({ text, color }: { text: string; color: string }) {
  return <span style={{ color }}>{text}</span>
}

// Method 2: Interface (preferred for shared/complex props)
interface ButtonProps {
  variant?:  'primary' | 'secondary' | 'ghost'
  size?:     'sm' | 'md' | 'lg'
  disabled?: boolean
  loading?:  boolean
  onClick?:  (event: React.MouseEvent<HTMLButtonElement>) => void
  children:  React.ReactNode
}

function Button({ variant = 'primary', size = 'md', children, ...props }: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant} btn-${size}`}
      {...props}
    >
      {children}
    </button>
  )
}

// Method 3: React.FC — not recommended (adds implicit children, poor inference)
const Button: React.FC<ButtonProps> = ({ children }) => <button>{children}</button>
```

<Callout type="warning" title="Avoid React.FC">
`React.FC` adds an implicit `children?: ReactNode` prop (even if you don't want it) and makes generic components harder to write. Prefer explicit prop types.
</Callout>

## Component Composition

```tsx
// Small, focused components compose into complex UIs

function Avatar({ src, alt, size = 'md' }: AvatarProps) {
  return <img src={src} alt={alt} className={`avatar avatar-${size}`} />
}

function UserCard({ user }: { user: User }) {
  return (
    <div className="user-card">
      <Avatar src={user.avatarUrl} alt={user.name} size="lg" />
      <div>
        <h3>{user.name}</h3>
        <p>{user.role}</p>
      </div>
    </div>
  )
}

function UserList({ users }: { users: User[] }) {
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>
          <UserCard user={user} />
        </li>
      ))}
    </ul>
  )
}
```

## Component Best Practices

| Practice | Why |
|---|---|
| Single responsibility | Easier to test, understand, reuse |
| Descriptive names | Self-documenting code |
| Props interface declared separately | Better IDE support, composable |
| Default props in destructuring | No need for external defaultProps |
| Keep JSX readable | Extract JSX into variables for complex conditional rendering |

## Summary

<Callout type="success" title="What we learned">
- A functional component is a function that returns JSX
- Use TypeScript interfaces for prop types (not `React.FC`)
- Keep components small and focused on one responsibility
- Compose simple components to build complex UIs
- Hooks must be called at the top level of the function body
</Callout>
