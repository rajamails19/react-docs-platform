# React Context API — Fundamentals

The React Context API solves a specific problem: passing data through a component tree without manually threading props through every intermediate component. This is called **prop drilling**, and it becomes painful when the data consumer is several levels below the data producer.

<Callout type="info">
Context is built into React — no additional libraries needed. It's the right tool for data that is "global" to a subtree: the current authenticated user, UI theme, active locale, feature flags, or current user preferences.
</Callout>

## The Prop Drilling Problem

Imagine a `currentUser` object that starts in `App` but is needed in the deeply nested `UserAvatar` component:

```tsx
// Without Context — props passed through every level
function App() {
  const user = { name: 'Alice', avatar: '/alice.jpg' }
  return <Layout user={user} />
}

function Layout({ user }: { user: User }) {
  return <Header user={user} />          // Layout doesn't need user, just passes it down
}

function Header({ user }: { user: User }) {
  return <Nav user={user} />              // Header doesn't need user either
}

function Nav({ user }: { user: User }) {
  return <UserAvatar user={user} />       // Nav doesn't need user
}

function UserAvatar({ user }: { user: User }) {
  return <img src={user.avatar} alt={user.name} />  // Finally used here
}
```

Every component in the chain must accept and pass `user` even though only `UserAvatar` uses it. With Context, `App` provides `user` once and `UserAvatar` reads it directly.

### 1. `createContext` — Define the Context

```tsx
import { createContext } from 'react'

// Provide a meaningful default value with correct TypeScript type
interface ThemeContextValue {
  theme: 'light' | 'dark'
  toggleTheme: () => void
}

// The default value is used when a component reads the context
// but has NO Provider above it in the tree.
const ThemeContext = createContext<ThemeContextValue>({
  theme: 'light',
  toggleTheme: () => {},
})
```

The default value in `createContext` serves two purposes:
1. It gives TypeScript the correct type without requiring `| null` everywhere.
2. It allows components to work in isolation (e.g., in Storybook or tests) without a Provider.

<Callout type="warning">
The default value is only used when a component consumes the context but has no matching Provider above it. If you wrap your app in a Provider (which you almost always do), the default value is never seen at runtime. Design it as a safe fallback, not as the real initial value.
</Callout>

### 2. `Context.Provider` — Supply the Value

```tsx
// contexts/ThemeContext.tsx
import { createContext, useState, useCallback } from 'react'
import type { FC, ReactNode } from 'react'

interface ThemeContextValue {
  theme: 'light' | 'dark'
  toggleTheme: () => void
}

export const ThemeContext = createContext<ThemeContextValue>({
  theme: 'light',
  toggleTheme: () => {},
})

export const ThemeProvider: FC<{ children: ReactNode }> = ({ children }) => {
  const [theme, setTheme] = useState<'light' | 'dark'>('light')

  const toggleTheme = useCallback(() => {
    setTheme((prev) => (prev === 'light' ? 'dark' : 'light'))
  }, [])

  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  )
}
```

The `value` prop on `Provider` is what consumers receive. Every time `value` changes (i.e., the provider re-renders with a new value), **all consumers re-render**. This is important for performance — we'll address it in the advanced patterns guide.

### 3. `useContext` — Consume the Value

```tsx
// components/ThemeToggle.tsx
import { useContext } from 'react'
import { ThemeContext } from '../contexts/ThemeContext'

export function ThemeToggle() {
  const { theme, toggleTheme } = useContext(ThemeContext)

  return (
    <button onClick={toggleTheme}>
      Switch to {theme === 'light' ? 'dark' : 'light'} mode
    </button>
  )
}

// components/AppShell.tsx
import { useContext } from 'react'
import { ThemeContext } from '../contexts/ThemeContext'

export function AppShell({ children }: { children: ReactNode }) {
  const { theme } = useContext(ThemeContext)

  return (
    <div className={`app-shell app-shell--${theme}`}>
      {children}
    </div>
  )
}
```

## Step-by-Step: Building an Auth Context

Here is a complete, production-ready auth context from scratch.

### Step 1: Define Types and Create the Context

```tsx
// contexts/AuthContext.tsx
import { createContext, useContext, useState, useCallback, useEffect } from 'react'
import type { FC, ReactNode } from 'react'

export interface User {
  id: string
  email: string
  name: string
  avatarUrl?: string
}

interface AuthContextValue {
  user: User | null
  isLoading: boolean
  isAuthenticated: boolean
  login: (email: string, password: string) => Promise<void>
  logout: () => void
}

// Using `null` as default forces consumers to be inside a Provider.
// We'll handle this in a custom hook below.
const AuthContext = createContext<AuthContextValue | null>(null)
```

### Step 2: Build the Provider Component

```tsx
// contexts/AuthContext.tsx (continued)
export const AuthProvider: FC<{ children: ReactNode }> = ({ children }) => {
  const [user, setUser] = useState<User | null>(null)
  const [isLoading, setIsLoading] = useState(true)

  // Restore session on mount
  useEffect(() => {
    const restoreSession = async () => {
      try {
        const token = localStorage.getItem('token')
        if (!token) return

        const response = await fetch('/api/me', {
          headers: { Authorization: `Bearer ${token}` },
        })

        if (response.ok) {
          setUser(await response.json())
        } else {
          localStorage.removeItem('token')
        }
      } finally {
        setIsLoading(false)
      }
    }
    restoreSession()
  }, [])

  const login = useCallback(async (email: string, password: string) => {
    const response = await fetch('/api/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    })
    if (!response.ok) throw new Error('Login failed')
    const { token, user: userData } = await response.json()
    localStorage.setItem('token', token)
    setUser(userData)
  }, [])

  const logout = useCallback(() => {
    localStorage.removeItem('token')
    setUser(null)
  }, [])

  return (
    <AuthContext.Provider
      value={{
        user,
        isLoading,
        isAuthenticated: user !== null,
        login,
        logout,
      }}
    >
      {children}
    </AuthContext.Provider>
  )
}
```

### Step 3: Build a Custom Hook

Always wrap `useContext` in a custom hook. It provides a better error message, hides the context object from consumers, and gives you a single import point.

```tsx
// contexts/AuthContext.tsx (continued)
export function useAuth(): AuthContextValue {
  const context = useContext(AuthContext)
  if (context === null) {
    throw new Error(
      'useAuth must be used within an <AuthProvider>. ' +
      'Wrap your component tree with <AuthProvider> in main.tsx or App.tsx.'
    )
  }
  return context
}
```

### Step 4: Wire the Provider Into Your App

```tsx
// main.tsx
import { createRoot } from 'react-dom/client'
import { BrowserRouter } from 'react-router-dom'
import { AuthProvider } from './contexts/AuthContext'
import { App } from './App'

createRoot(document.getElementById('root')!).render(
  <BrowserRouter>
    <AuthProvider>
      <App />
    </AuthProvider>
  </BrowserRouter>
)
```

Provider order matters. `BrowserRouter` is outside so routing works everywhere, including inside `AuthProvider`. If `AuthProvider` needed routing hooks, it would need to be inside `BrowserRouter`.

### Step 5: Consume in Any Descendant Component

```tsx
// components/UserMenu.tsx
import { Link } from 'react-router-dom'
import { useAuth } from '../contexts/AuthContext'

export function UserMenu() {
  const { user, isAuthenticated, logout } = useAuth()

  if (!isAuthenticated) {
    return <Link to="/login">Sign in</Link>
  }

  return (
    <div className="user-menu">
      <img src={user!.avatarUrl} alt={user!.name} />
      <span>{user!.name}</span>
      <button onClick={logout}>Sign out</button>
    </div>
  )
}
```

<Callout type="info">
The core question: does this data need to be shared across components that don't have a direct parent-child relationship? If yes, and if it doesn't change more than a few times per second, Context is likely the right tool.
</Callout>

## Multiple Contexts in One App

Apps commonly use several independent contexts. Each context manages a focused concern.

```tsx
// main.tsx
import { createRoot } from 'react-dom/client'
import { BrowserRouter } from 'react-router-dom'
import { QueryClientProvider, QueryClient } from '@tanstack/react-query'
import { AuthProvider } from './contexts/AuthContext'
import { ThemeProvider } from './contexts/ThemeContext'
import { NotificationProvider } from './contexts/NotificationContext'
import { App } from './App'

const queryClient = new QueryClient()

createRoot(document.getElementById('root')!).render(
  <BrowserRouter>
    <QueryClientProvider client={queryClient}>
      <AuthProvider>
        <ThemeProvider>
          <NotificationProvider>
            <App />
          </NotificationProvider>
        </ThemeProvider>
      </AuthProvider>
    </QueryClientProvider>
  </BrowserRouter>
)
```

Each Provider is independent — `ThemeProvider` doesn't know about `AuthProvider`. Components consume only what they need. This separation prevents unrelated state changes from causing unnecessary re-renders across the whole app.

<Callout type="tip">
If your `main.tsx` starts looking like a pyramid of providers, consider a `AppProviders` wrapper component that composes them all. This keeps `main.tsx` clean without changing the behavior.
</Callout>

```tsx
// providers/AppProviders.tsx
import type { FC, ReactNode } from 'react'
import { QueryClientProvider, QueryClient } from '@tanstack/react-query'
import { AuthProvider } from '../contexts/AuthContext'
import { ThemeProvider } from '../contexts/ThemeContext'

const queryClient = new QueryClient()

export const AppProviders: FC<{ children: ReactNode }> = ({ children }) => (
  <QueryClientProvider client={queryClient}>
    <AuthProvider>
      <ThemeProvider>
        {children}
      </ThemeProvider>
    </AuthProvider>
  </QueryClientProvider>
)

// main.tsx — now clean
createRoot(document.getElementById('root')!).render(
  <BrowserRouter>
    <AppProviders>
      <App />
    </AppProviders>
  </BrowserRouter>
)
```

## Context with `useReducer` for Complex State

For state with multiple sub-values and complex update logic, combine Context with `useReducer`. This is the built-in alternative to Redux for moderate complexity.

```tsx
// contexts/CartContext.tsx
import { createContext, useContext, useReducer, useCallback } from 'react'
import type { FC, ReactNode } from 'react'

interface CartItem {
  id: string
  name: string
  price: number
  quantity: number
}

interface CartState {
  items: CartItem[]
  isOpen: boolean
}

type CartAction =
  | { type: 'ADD_ITEM'; payload: Omit<CartItem, 'quantity'> }
  | { type: 'REMOVE_ITEM'; payload: { id: string } }
  | { type: 'UPDATE_QUANTITY'; payload: { id: string; quantity: number } }
  | { type: 'CLEAR_CART' }
  | { type: 'TOGGLE_CART' }

function cartReducer(state: CartState, action: CartAction): CartState {
  switch (action.type) {
    case 'ADD_ITEM': {
      const existing = state.items.find((i) => i.id === action.payload.id)
      if (existing) {
        return {
          ...state,
          items: state.items.map((i) =>
            i.id === action.payload.id ? { ...i, quantity: i.quantity + 1 } : i
          ),
        }
      }
      return { ...state, items: [...state.items, { ...action.payload, quantity: 1 }] }
    }
    case 'REMOVE_ITEM':
      return { ...state, items: state.items.filter((i) => i.id !== action.payload.id) }
    case 'UPDATE_QUANTITY':
      return {
        ...state,
        items: state.items.map((i) =>
          i.id === action.payload.id ? { ...i, quantity: action.payload.quantity } : i
        ),
      }
    case 'CLEAR_CART':
      return { ...state, items: [] }
    case 'TOGGLE_CART':
      return { ...state, isOpen: !state.isOpen }
    default:
      return state
  }
}

interface CartContextValue {
  state: CartState
  dispatch: React.Dispatch<CartAction>
  totalItems: number
  totalPrice: number
}

const CartContext = createContext<CartContextValue | null>(null)

export const CartProvider: FC<{ children: ReactNode }> = ({ children }) => {
  const [state, dispatch] = useReducer(cartReducer, { items: [], isOpen: false })

  const totalItems = state.items.reduce((sum, item) => sum + item.quantity, 0)
  const totalPrice = state.items.reduce((sum, item) => sum + item.price * item.quantity, 0)

  return (
    <CartContext.Provider value={{ state, dispatch, totalItems, totalPrice }}>
      {children}
    </CartContext.Provider>
  )
}

export function useCart(): CartContextValue {
  const context = useContext(CartContext)
  if (!context) throw new Error('useCart must be used within a CartProvider')
  return context
}
```

```tsx
// Using the cart context
export function AddToCartButton({ product }: { product: { id: string; name: string; price: number } }) {
  const { dispatch } = useCart()

  return (
    <button onClick={() => dispatch({ type: 'ADD_ITEM', payload: product })}>
      Add to Cart
    </button>
  )
}

export function CartBadge() {
  const { totalItems } = useCart()
  return <span className="badge">{totalItems}</span>
}
```

## Summary

- **`createContext(defaultValue)`** — creates a context object. The default is used only when no Provider is above the consumer.
- **`Context.Provider value={...}`** — wraps a subtree and makes `value` available to all descendants. Re-renders consumers when `value` changes.
- **`useContext(MyContext)`** — reads the nearest Provider's value. Wrap in a custom hook for better DX and error messages.
- Build context in four steps: create → provider component → custom hook → mount provider in your app tree.
- Use Context for infrequently-changing, globally-needed data: auth, theme, locale, notifications.
- Avoid Context for high-frequency updates, component-local state, or complex state that benefits from dedicated tools.
- Multiple independent contexts is the right pattern — one context per concern, not one giant context for everything.
