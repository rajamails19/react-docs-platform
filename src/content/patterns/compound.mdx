# Compound Components

The **Compound Components** pattern creates a group of related components that work together through implicit shared state. It's how UI libraries like Radix UI, Headless UI, and Reach UI are built — and what separates a great component API from a rigid one.

## The Problem: Prop Hell

```tsx
// ❌ Everything crammed into one component
<Select
  options={options}
  value={value}
  onChange={onChange}
  placeholder="Select..."
  searchable
  clearable
  renderOption={(opt) => <CustomOption opt={opt} />}
  renderValue={(val) => <CustomValue val={val} />}
  groupBy="category"
  maxHeight={300}
  loading={isLoading}
  emptyMessage="No results"
  footer={<a href="/add">+ Add new</a>}
  onOpen={handleOpen}
  onClose={handleClose}
  // ...15 more props
/>
```

Every new use case requires a new prop. The API becomes unpredictable and the component becomes impossible to customize without forking.

## The Solution: Compound Components

```tsx
// ✅ Composable API — consumers control the structure
<Select value={value} onChange={onChange}>
  <Select.Trigger placeholder="Select an option..." />
  <Select.Content>
    <Select.Search />
    {options.map(opt => (
      <Select.Option key={opt.value} value={opt.value}>
        <Icon name={opt.icon} />
        {opt.label}
      </Select.Option>
    ))}
    <Select.Empty>No results found</Select.Empty>
    <Select.Footer>
      <a href="/add">+ Add item</a>
    </Select.Footer>
  </Select.Content>
</Select>
```

The consumer owns the structure. The components share state through Context.

## Implementation: Tabs Component

```tsx
// 1. Define the shared state interface
interface TabsContextValue {
  activeTab: string
  setActiveTab: (id: string) => void
}

// 2. Create Context
const TabsContext = createContext<TabsContextValue | null>(null)

// 3. Create a hook for children to access shared state
function useTabsContext() {
  const ctx = useContext(TabsContext)
  if (!ctx) throw new Error('Tabs compound components must be used within <Tabs>')
  return ctx
}

// 4. Root component — owns the state
function Tabs({
  children,
  defaultTab,
  onChange,
}: {
  children: React.ReactNode
  defaultTab: string
  onChange?: (tab: string) => void
}) {
  const [activeTab, setActiveTab] = useState(defaultTab)

  const handleChange = (id: string) => {
    setActiveTab(id)
    onChange?.(id)
  }

  return (
    <TabsContext.Provider value={{ activeTab, setActiveTab: handleChange }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  )
}

// 5. Sub-components consume shared state via Context
function TabList({ children }: { children: React.ReactNode }) {
  return <div role="tablist" className="tab-list">{children}</div>
}

function Tab({ id, children }: { id: string; children: React.ReactNode }) {
  const { activeTab, setActiveTab } = useTabsContext()
  const isActive = activeTab === id

  return (
    <button
      role="tab"
      aria-selected={isActive}
      aria-controls={`panel-${id}`}
      onClick={() => setActiveTab(id)}
      className={`tab ${isActive ? 'tab-active' : ''}`}
    >
      {children}
    </button>
  )
}

function TabPanels({ children }: { children: React.ReactNode }) {
  return <div className="tab-panels">{children}</div>
}

function TabPanel({ id, children }: { id: string; children: React.ReactNode }) {
  const { activeTab } = useTabsContext()
  if (activeTab !== id) return null

  return (
    <div role="tabpanel" id={`panel-${id}`} className="tab-panel">
      {children}
    </div>
  )
}

// 6. Attach sub-components as static properties
Tabs.List   = TabList
Tabs.Tab    = Tab
Tabs.Panels = TabPanels
Tabs.Panel  = TabPanel
```

## Using the Compound Component

```tsx
// Clean, readable, flexible consumer API
function SettingsPage() {
  return (
    <Tabs defaultTab="profile" onChange={tab => analytics.track('tab_change', { tab })}>
      <Tabs.List>
        <Tabs.Tab id="profile">
          <UserIcon size={14} /> Profile
        </Tabs.Tab>
        <Tabs.Tab id="security">
          <ShieldIcon size={14} /> Security
        </Tabs.Tab>
        <Tabs.Tab id="billing">
          <CreditCardIcon size={14} /> Billing
        </Tabs.Tab>
      </Tabs.List>

      <Tabs.Panels>
        <Tabs.Panel id="profile">
          <ProfileForm />
        </Tabs.Panel>
        <Tabs.Panel id="security">
          <SecuritySettings />
        </Tabs.Panel>
        <Tabs.Panel id="billing">
          <BillingDetails />
        </Tabs.Panel>
      </Tabs.Panels>
    </Tabs>
  )
}
```

## When to Use Compound Components

| Scenario | Good fit? |
|---|---|
| Complex UI with many configuration options | ✅ Yes |
| Consumer needs control over structure/layout | ✅ Yes |
| Multiple sub-components share state | ✅ Yes |
| Simple button or badge | ❌ Overkill |
| State has no sub-components | ❌ Not applicable |
| Component library being shipped to other teams | ✅ Excellent choice |

## Summary

<Callout type="success" title="What we learned">
- Compound Components share state implicitly through Context, eliminating prop drilling
- The pattern gives consumers **full control over structure** while hiding state complexity
- It scales perfectly for component libraries where consumers have diverse layout needs
- The `throw new Error` in the Context hook provides a clear DX when misused
- Libraries like Radix UI, Headless UI, and Reach UI are all built on this pattern
</Callout>
