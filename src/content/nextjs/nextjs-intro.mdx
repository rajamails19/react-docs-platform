# Introduction to Next.js

Next.js is the most widely adopted **React meta-framework** in production. It takes React — a UI library — and wraps it with everything you need to ship a real application: server-side rendering, static generation, file-based routing, API routes, image optimization, font loading, middleware, and edge functions. As of 2024, it powers the frontends of Vercel, TikTok, Twitch, Nike, Hulu, and thousands of other production applications.

<Callout type="info">
A **meta-framework** builds on top of another framework and makes opinionated decisions so you don't have to. Next.js is to React what Rails is to Ruby — it provides structure, conventions, and built-in solutions for cross-cutting concerns.
</Callout>

## What Problems Does Next.js Solve?

Plain React (via Create React App or Vite) gives you a client-side single-page application. That's powerful for interactivity, but it comes with a set of fundamental problems that every serious application eventually hits:

**1. SEO and Social Sharing**
A blank `<div id="root"></div>` is what crawlers and social media link previewers see when they visit your React SPA. Google's crawler eventually executes JavaScript, but it's slower and less reliable than receiving fully rendered HTML. Twitter, LinkedIn, and Slack's link unfurlers don't execute JS at all — they read raw HTML for Open Graph tags.

**2. Performance on First Load**
The browser must download your entire JavaScript bundle before rendering anything meaningful. On slow connections or low-end devices, users see a white screen for seconds. The Time to First Byte (TTFB) and First Contentful Paint (FCP) metrics suffer.

**3. No Server — No Database Access at Render Time**
A React SPA can only fetch data after it has mounted in the browser. Every page load requires a network round trip: load JS → render loading skeleton → fetch data → render UI. Next.js allows components to fetch data at render time on the server, eliminating the waterfall entirely.

**4. Routing Boilerplate**
React has no built-in router. Every project installs React Router, configures routes, handles 404s, and sets up nested layouts manually. Next.js handles this through the file system.

**5. API Routes**
SPAs need a separate backend server for any server-side logic (auth, database writes, secrets). Next.js lets you write API handlers in the same repository, even in the same file system.

**6. Image and Font Optimization**
Serving large, unoptimized images is one of the most common causes of poor Core Web Vitals. Next.js's `<Image>` component automatically resizes, converts to WebP, lazy-loads, and prevents layout shift. Fonts are subset and self-hosted with zero layout shift by default.

<Callout type="tip">
The rendering model comparison deserves special attention. **CSR** means the browser does all the work. **SSR** means the server renders HTML on every request. **SSG** means HTML is generated once at build time. **ISR** (Incremental Static Regeneration) means HTML is generated at build time but can be refreshed on a schedule or on-demand. Next.js lets you choose per route — or even per fetch call.
</Callout>

Next.js has two routing systems. Understanding both is essential because most large codebases still use Pages Router and are migrating incrementally.

<Callout type="warning">
Both routers can coexist in the same project during migration. A file in `app/` and a file in `pages/` can serve different routes simultaneously. However, the same route cannot exist in both directories.
</Callout>

## Creating a Next.js Project

```bash
npx create-next-app@latest my-app
```

The CLI will ask a series of questions:

```
✔ Would you like to use TypeScript? › Yes
✔ Would you like to use ESLint? › Yes
✔ Would you like to use Tailwind CSS? › Yes
✔ Would you like to use `src/` directory? › Yes
✔ Would you like to use App Router? (recommended) › Yes
✔ Would you like to customize the default import alias (@/*)? › No
```

<Callout type="tip">
Always choose TypeScript and App Router for new projects in 2024. The `src/` directory is optional but keeps your source files separate from config files at the root.
</Callout>

## Key Concepts at a Glance

### Server Components (Default)

In the App Router, every component is a Server Component by default. They run exclusively on the server — never in the browser. This means:

- They can `await` database calls, file reads, or any async operation directly
- They never ship their source code to the client
- They cannot use `useState`, `useEffect`, or browser APIs
- They reduce client-side JavaScript to zero for entire subtrees

```tsx
// app/posts/page.tsx — This is a Server Component
// No 'use client' directive = server-only

async function PostsPage() {
  // Direct database access — no useEffect, no loading state
  const posts = await db.query('SELECT * FROM posts ORDER BY created_at DESC')

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>
          <a href={`/posts/${post.slug}`}>{post.title}</a>
        </li>
      ))}
    </ul>
  )
}

export default PostsPage
```

### Client Components

Adding `'use client'` at the top of a file marks it and all its imports as Client Components. They work exactly like React components you know from Vite/CRA: they run in the browser and can use all hooks and browser APIs.

```tsx
'use client'
// app/components/SearchBar.tsx

import { useState } from 'react'

export function SearchBar({ onSearch }: { onSearch: (q: string) => void }) {
  const [query, setQuery] = useState('')

  return (
    <input
      value={query}
      onChange={e => {
        setQuery(e.target.value)
        onSearch(e.target.value)
      }}
      placeholder="Search..."
    />
  )
}
```

### Layouts

A `layout.tsx` file wraps all pages at the same directory level and below. The root layout is mandatory and must include `<html>` and `<body>` tags.

```tsx
// app/layout.tsx — Root layout
import type { Metadata } from 'next'
import { Inter } from 'next/font/google'

const inter = Inter({ subsets: ['latin'] })

export const metadata: Metadata = {
  title: { template: '%s | My App', default: 'My App' },
  description: 'My awesome application',
}

export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html lang="en">
      <body className={inter.className}>
        <nav>Global Navigation</nav>
        {children}
        <footer>Global Footer</footer>
      </body>
    </html>
  )
}
```

### loading.tsx and Suspense

Placing a `loading.tsx` file next to a `page.tsx` automatically wraps that page in a `<Suspense>` boundary. While the Server Component is fetching data, the loading UI is shown immediately — enabling streaming.

```tsx
// app/posts/loading.tsx
export default function PostsLoading() {
  return (
    <div className="space-y-4">
      {Array.from({ length: 5 }).map((_, i) => (
        <div key={i} className="h-16 bg-gray-200 rounded animate-pulse" />
      ))}
    </div>
  )
}
```

### error.tsx and Error Boundaries

An `error.tsx` file automatically becomes an error boundary for its route segment. It must be a Client Component (errors are caught during rendering, which can happen client-side).

```tsx
'use client'
// app/posts/error.tsx

import { useEffect } from 'react'

export default function PostsError({
  error,
  reset,
}: {
  error: Error & { digest?: string }
  reset: () => void
}) {
  useEffect(() => {
    console.error('Posts error:', error)
  }, [error])

  return (
    <div className="text-center py-12">
      <h2 className="text-xl font-semibold text-red-600">
        Failed to load posts
      </h2>
      <p className="text-gray-500 mt-2">{error.message}</p>
      <button
        onClick={reset}
        className="mt-4 px-4 py-2 bg-blue-600 text-white rounded"
      >
        Try again
      </button>
    </div>
  )
}
```

## When to Use Next.js vs a React SPA

<Callout type="info">
This is one of the most common architectural questions in React development. Neither is universally better — it depends on your requirements.
</Callout>

**Choose Next.js when:**
- SEO matters (marketing pages, blogs, e-commerce, documentation)
- You need data at render time (dashboards with real-time data, product pages)
- You want to ship a full-stack application from one repository
- You need ISR for high-traffic pages with semi-static content
- Performance metrics (FCP, LCP) are important for your audience
- Your team is building a long-term product that will grow

**Choose a React SPA (Vite) when:**
- Building a highly interactive tool that is always authenticated (Figma-like, internal dashboards)
- The app is completely behind login (SEO is irrelevant)
- You already have a dedicated backend and just need a UI
- You need maximum flexibility in your deployment target (pure static CDN)
- Building a prototype where setup speed matters more than architecture

## Next.js 14 Features Overview

Next.js 14 (released October 2023) is the current recommended version with these major features:

**Turbopack (Stable for Development)**
The Rust-based bundler that replaces Webpack in development. Up to 53% faster local server startup and 94% faster code updates. Enable it with `next dev --turbo`.

**Server Actions (Stable)**
Write server-side mutation functions that can be called directly from Client Components — no separate API route needed. They work with standard HTML forms and `action` attributes.

```tsx
// app/actions.ts
'use server'

import { revalidatePath } from 'next/cache'

export async function createPost(formData: FormData) {
  const title = formData.get('title') as string
  await db.insert({ title })
  revalidatePath('/posts')
}
```

```tsx
// app/new-post/page.tsx
import { createPost } from '../actions'

export default function NewPostPage() {
  return (
    <form action={createPost}>
      <input name="title" placeholder="Post title" />
      <button type="submit">Create Post</button>
    </form>
  )
}
```

**Partial Prerendering (Preview)**
A new rendering model that combines SSG speed with SSR dynamism. Static shell is served from the CDN immediately; dynamic holes stream in from the server. Zero configuration changes required — just use `<Suspense>`.

**Metadata API Improvements**
Static and dynamic metadata with full TypeScript support, including Open Graph images generated at runtime with `opengraph-image.tsx`.

## Deployment

Next.js's first-class deployment target is **Vercel** (the company that created Next.js). Deploying is as simple as pushing to GitHub with the Vercel integration. Features like ISR, Edge Middleware, Edge Functions, Image Optimization, and Partial Prerendering work out of the box.

**Self-hosting options:**
- **Node.js server:** `next build && next start` — works anywhere Node.js runs
- **Docker:** Official Dockerfile in the Next.js examples repository
- **Static export:** `output: 'export'` in next.config.js — no server required, but disables SSR, ISR, Route Handlers, and Image Optimization
- **Other platforms:** AWS Amplify, Railway, Render, Fly.io, and Netlify all have Next.js support

<Callout type="warning">
If you self-host, you are responsible for replicating features that Vercel handles automatically: caching headers for ISR, image optimization CDN, edge function distribution, and automatic scaling. These are non-trivial to replicate correctly.
</Callout>
