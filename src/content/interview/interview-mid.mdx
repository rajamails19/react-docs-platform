# Mid-Level React Interview Questions

These questions test whether you understand React deeply — not just how to use it, but why it works the way it does. Expect these in 2-5 year experience roles.

## Performance

## State Management

<InterviewQA questions={[
  {
    question: "When would you choose Redux vs Context API vs React Query?",
    answer: "Context API: best for global UI state that rarely changes — theme, auth user, locale. Re-renders all consumers on change, so don't use for frequently updating data. Redux/Zustand: best for complex shared state with many updates, time-travel debugging needs, or when multiple disconnected components need to update the same state. React Query/SWR: best for server state — data that lives on a server, needs caching, background refetching, optimistic updates, or synchronization. The key insight: most state management problems are actually server state problems — React Query solves them better than Redux.",
    level: "mid"
  },
  {
    question: "What is the difference between server state and client state?",
    answer: "Server state is data that lives on a server and is fetched over the network — it's asynchronous, can be stale, shared with other users, and needs caching/synchronization. Client state is local UI state that exists only in the browser — toggle open/closed, form input values, selected tab. The common mistake is managing server state with useState/useEffect — you end up reinventing caching, deduplication, background refetching, and error handling. React Query and SWR are built specifically for server state. Redux is for client state (or was designed for a time when the distinction wasn't clear).",

## React Internals

<InterviewQA questions={[
  {
    question: "What is reconciliation and how does React's diffing algorithm work?",
    answer: "Reconciliation is React's process of updating the DOM efficiently. When state changes, React renders a new component tree and compares it to the previous tree (diffing). React's diffing heuristics: (1) Different element types — React tears down the old tree and builds a new one from scratch. (2) Same element type — React updates the existing DOM node's attributes. (3) Lists — uses key props to match old and new items. React assumes that components at the same position in the tree are the same type across renders, so swapping components by type at the same level triggers a full remount. Full tree comparison is O(n³), React's heuristics reduce it to O(n).",
    level: "mid"
  },
  {
    question: "What is the useLayoutEffect hook and when do you need it?",
    answer: "useLayoutEffect is identical to useEffect but fires synchronously after DOM mutations and before the browser paints. Use it when you need to read DOM measurements (getBoundingClientRect, offsetWidth) or mutate the DOM before the user sees anything — to prevent visual flicker. Example: a tooltip that needs to know its position relative to the trigger element must use useLayoutEffect to measure and position itself before paint. For everything else (data fetching, subscriptions, timers), use useEffect — it runs asynchronously and doesn't block painting.",

## Summary

<Callout type="success" title="Key Takeaways for Mid-Level Interviews">
- Understand why components re-render — state change, parent re-render, context change
- Know when to use React.memo, useMemo, useCallback — and when NOT to (premature optimization)
- Distinguish server state (React Query) from client state (Context/Zustand/Redux)
- Be able to describe a feature-based folder structure and explain the tradeoffs
- Know the difference between useEffect (async, after paint) and useLayoutEffect (sync, before paint)
</Callout>
